# Contributing to Scaffel

Thank you for your interest in contributing to Scaffel! This document provides guidelines and instructions for contributing.

---

## Getting Started

1. **Fork the repository** and clone your fork locally
2. **Create a branch** for your contribution: `git checkout -b feature/your-feature-name`
3. **Read the documentation** in the [`docs/`](./docs/) folder to understand the architecture
4. **Follow the development setup** (see [Action Plan](./docs/ACTION-PLAN.md))

---

## Development Guidelines

### Code Style

- Follow existing code patterns and conventions
- Write clear, self-documenting code
- Add comments for complex logic
- Keep functions focused and single-purpose

### Testing

- Write tests for new features
- Ensure all existing tests pass
- Add integration tests for CLI commands
- Test error cases and edge conditions

### Documentation

- Update relevant documentation when adding features
- Add examples for new CLI commands
- Update API reference if adding new programmatic APIs
- Keep README.md and docs/ up to date

---

## Contribution Process

1. **Check existing issues** - See if your idea is already being worked on
2. **Create an issue** - For significant changes, discuss first in an issue
3. **Make your changes** - Follow the development guidelines above
4. **Test thoroughly** - Ensure your changes work as expected
5. **Submit a pull request** - Include a clear description of changes

---

## Pull Request Guidelines

### PR Title Format

Use clear, descriptive titles:
- `feat: Add roadmap export functionality`
- `fix: Resolve dependency resolution bug`
- `docs: Update API reference examples`
- `refactor: Simplify code generator logic`

### PR Description

Include:
- **What** - What does this PR change?
- **Why** - Why is this change needed?
- **How** - How does it work?
- **Testing** - How was it tested?

### Code Review

- Address review feedback promptly
- Keep PRs focused and reasonably sized
- Update documentation as needed
- Ensure CI checks pass

---

## Areas for Contribution

### High Priority

- Core roadmap generation logic
- Code scaffold templates
- CLI command improvements
- Documentation improvements
- Test coverage

### Feature Ideas

- Additional template types
- Integration with other tools
- Performance optimizations
- Better error messages
- Enhanced validation

---

## Questions?

- Open an issue for questions or discussions
- Check existing documentation in [`docs/`](./docs/)
- Review the [Action Plan](./docs/ACTION-PLAN.md) for implementation details

---

## Code of Conduct

- Be respectful and inclusive
- Provide constructive feedback
- Help others learn and grow
- Focus on what's best for the project
